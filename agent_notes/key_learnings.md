# Key Learnings

Dense memory bank for surprising lessons learned by trying things. Keep this
file under 200 lines by recompressing it. Raw chronology belongs in
`agent_notes/loose_notes/`.

- `uv runpython` is just a typo, but `uv run <script.py>` can fail differently from `uv run python <script.py>` when the active directory/project is wrong. The durable fix was making `dynaworld` its own `uv` project and running from that project root.
- `uv` already has a shared global package cache by default. A cache env var is only needed to override the default location, not to make sharing happen at all.
- MPS made the tiny TokenGS experiments usable but not automatically fast. Kernel launch/resource contention from another tab can dominate perceived trainer speed.
- Lowering token count is a speed knob, but it also changes representational capacity. Decoupling encoder input size from render/loss size was the better smoke-test knob because it preserved the model input contract while cutting raster cost.
- The naive tiled renderer with nested Python loops was slower than dense PyTorch for the small Mac test case. Tile-based rasterization only pays off when culling/sorting is implemented cheaply, not when Python loops become the scheduler.
- A one-big-tile path being faster than expected showed that "tiled vs dense" was not the real binary. The overhead and code path mattered as much as the raster math.
- For the current renderer, DUSt3R lens metadata beyond pinhole intrinsics/extrinsics is noise. F-stop, lens type, shutter, and sensor metadata do not help unless we add distortion or photometric modeling.
- DUSt3R appeared to run "two passes," but the second phase was global alignment, not duplicate inference. Future logs should label pair inference vs alignment more clearly.
- Stale extracted frame folders are dangerous UX even when the code loads explicit JSON records. Users look at folders, so extraction should clear or isolate outputs to avoid misleading evidence.
- Rendering without camera extrinsics lets the model cheat with frame-conditioned gaussians. For dynamic novel view synthesis, using `camera_to_world` in the renderer is not optional.
- DUSt3R per-frame focal estimates can jitter on a phone clip. Median focal across the sequence is often a better default than trusting each frame.
- A single shared `BaseTrainer` would hide real differences between known-camera, image-implicit, and video-token implicit training. Shared payload contracts are cleaner than shared trainer inheritance.
- Positional tuple outputs were fine for toy iteration but became risky as soon as camera state, videos, and multiple baselines appeared. Named runtime payloads are the right boundary.
- `SequenceData.__getitem__` was useful as a temporary migration crutch, but it hid weak contracts. After trainer call sites moved to attributes and `ClipBatch`, deleting the bridge made violations easy to grep.
- System `python3` can pass compile checks but fail import smokes because dependencies live in the `uv` environment. Use `uv run python` for meaningful Dynaworld checks.
- The smoke wrapper lacked execute permission even though the full wrappers were executable. Always check script mode before assuming a shell script can be run with `./...`.
- JSONC configs are better than environment-variable fanout for trainer experiments. Shell wrappers should choose config files rather than re-declaring every knob.
- Shared JSONC helpers belong in one tiny utility. Keeping `strip_jsonc_comments(...)`, config loading, and W&B-safe serialization in `src/train/config_utils.py` avoids both parser copy-paste and a mirrored Python default map.
- `render_size` should be treated as a viewport, not a crop. Scale `fx/fy/cx/cy` from encoder input pixels to render pixels at the render boundary, then resize GT for loss; pose and FoV stay unchanged.
- Keep full experiment defaults in JSONC, not in a mirrored Python `DEFAULT_CONFIG`. Python should normalize runtime values, validate required sections, and keep constructor-name mapping in one boundary factory.
- On the MPS prebaked-camera baseline, AMP/flash attention was slower because dense splat render/backward dominated. Batched dense rendering improved frame throughput more than reducing `frames_per_step`, which mainly improved iteration rate.
- Do not synthesize higher-FPS camera-bake videos from an already downsampled low-FPS clip. The duplicate/interpolated temporal signal can distort DUSt3R camera estimates; generate bake variants from the original source frames with the final crop/FPS/scale in one ffmpeg pass.
- DUSt3R pose scale is arbitrary enough to break fixed-volume Gaussian heads. The 128px/4fps bake produced camera z translation up to `2.56` while TokenGS initial z is constrained to `[0.5, 2.5]`, putting late-frame Gaussians behind the camera and causing render NaNs before any optimizer update.
- The committed stable prebaked-camera baseline was 32px/2fps/all-frames, not the new 128px/4fps camera trajectory. Treat 64/128 video-size bakes as new camera-distribution tests, not just higher-resolution versions of the same stable run.
- Camera/scene normalization is a pretraining contract, not just a local init tweak. Gaussian extent, camera radius, path residual scale, FoV, near plane, and Plucker moment scale all need a shared canonical frame or each DUSt3R solve can define a different task.
- Upstream `taichi-splatting==0.13.0` is not installable in the repo's Python 3.11 `uv` env, but the vendored fork now has a Taichi 1.7.4/Metal-compatible Torch autograd path. Treat the local fork as the source of truth for Mac renderer experiments, not the pip package.
- The copied `memory_efficient_sparse` rasterizer is not a good MPS benchmark default despite its name. Its per-splat Python-loop custom autograd path was hundreds of ms at 64px/128 splats and seconds at 64px/512 splats, while custom/vectorized sparse stayed around tens of ms after warmup.
- Benchmark summaries must account for skipped optional backends. Timing means over only `status == ok` silently hid Taichi/gsplat on MPS, making the renderer matrix look smaller than what actually ran.
- The checked-in trainers are still tiny for renderer scaling: the prebaked-camera defaults decode 128 tokens x 4 splats/token = 512 splats. A 64k-splat real-time target needs a separate forward-only throughput benchmark; dense `G*H*W` paths are not meaningful at that scale.
- Porting `taichi-splatting` to Mac is not a simple CUDA-shader swap. CUDA sort/cumsum can be replaced with Torch fallback, but the fast rasterizer also uses Taichi SIMT block/warp intrinsics unsupported by Metal. A Metal path needs separate forward/backward kernel structure; the current local patch is a simple global-memory reference path, not a tuned Metal renderer.
- Taichi/Metal benchmark timing needs an explicit Taichi sync at timing boundaries (`TaichiQueue.run_sync(ti.sync)`), not just `torch.mps.synchronize()`. Without that hook, mixed PyTorch/Taichi measurements are easy to mistrust even when the final numbers happen to stay similar after sync.
- The local Taichi/Metal backward path is not just "finite gradients": against a direct CPU-float64 Torch packed-2D raster reference, 16/32px cases with 4/8/16 splats matched forward and packed/features gradients at ~1e-8 absolute error. This validates the core simple Metal raster/backward math, separate from 3D projection correctness or performance.
- Taichi 1.7.4's public `parallel_sort` works on Taichi fields, not Torch tensors, and `PrefixSumExecutor` explicitly rejects Metal. A native Metal bin/sort path cannot just flip the current Torch fallback to Taichi algorithms; custom ndarray scan/sort kernels work on MPS but odd-even sort is far too slow (`128x128/G=65536` around 1.5s).
- A Metal-native ordered per-tile binning path that preserves input order is correct when splats are already depth-sorted, but the naive tile-major implementation is still slower than the Torch-sort reference (`128x128/G=65536` 68ms vs 31ms). The next real win needs a better algorithm, not merely “move sort to Metal.”
- A Taichi-only compact bucket sorter can do the right high-level GPU-binning shape on Metal and passes forward/backward accuracy, but it still loses to the Torch-sort reference (`1024x1024/G=65536` 125.6ms vs 52.1ms forward). Without a fused threadgroup-memory sort+raster kernel, Taichi spends too much time in separate bin/sort/raster stages.
- Low precision is not a simple cast in the Taichi path: the current tile mapper imports f32 `Gaussian2D` and expects f32 depths. fp16 needs separate mapper/raster kernels, not just `features.to(float16)`.
- MLX `mx.fast.metal_kernel(name=...)` requires a valid Metal/C identifier. The raw rasterizer's first cache-derived names contained `=`, `.`, and other config punctuation, causing Metal compile failure before any math ran.
- MLX `mx.eval(...)` is an in-place evaluator in the tested `mlx==0.31.1` path and returns `None`; evaluate first, then read `.item()` from the original array.
- Raw MLX/Metal and the Taichi accuracy reference sample pixels at `x + 0.5`, while the current Dynaworld dense/Torch sparse training renderers sample integer grid coordinates. Use the packed-2D accuracy benchmark for correctness; throughput comparisons against training renderers can show expected image diffs until sampling conventions are unified.
- Torch custom-op schemas declared as fixed tuple returns, e.g. `-> (Tensor, Tensor)`, must be implemented with fixed C++ tuple return types. Returning `std::vector<torch::Tensor>` registers as `Tensor[]` and aborts at extension import with a schema mismatch.
- Do not pass tightly packed Torch `[G,3]` float tensors to Metal kernels as `device float3*`. Metal `float3` has 16-byte alignment/stride while Torch rows are 12 bytes, so splats after index 0 read corrupted conics/colors; use flat `device float*` plus `index * 3` loads or an explicitly packed type.
- Random-splat sort pressure is not currently dominated by Taichi's OBB tile test: exact conic/tile intersection only reduced keys by ~8-10% at 64px/1024 and 128-1024px/64k. Same-tile custom rectangular bounds were ~1.29x exact keys in a 64px/512 probe, so future speed work should check `K/G` before assuming AccuTile will fix Taichi.
- In the Torch+Metal fastpath, saving sorted tile IDs fixed one real backward tax, but v3 still beat v2 while re-sorting in fast backward. That shifts the next optimization model toward parameter bandwidth and global atomic gradient pressure, especially under tile overlap.
- Porting saved sorted tile IDs into v3 confirmed the sort tax is still worth removing: it made forward ~1-2ms slower at 4K/64K because of writeback, but reduced estimated backward ~12-14% and forward+backward ~7-9% in no-overflow synthetic cases.
- Direct Torch projected splatting is only a small-scene baseline, not a 4K/64K comparator. At 128x128/512 it was already ~22-27x slower than v3 forward and ~73-93x slower forward+backward; 4K/64K would be ~1.1e12 pixel-splat evaluations.
- Passing MPS slice views like `packed[b]` through the single-image Taichi autograd function can trip PyTorch version-counter checks during backward. Native Taichi batch rendering avoids that train-loop hazard by keeping `[B, G, ...]` tensors intact through one autograd call.
- Small low-res/low-splat Mac raster benchmarks are launch/overhead dominated, so v2 can beat v3 there even though v3 wins large-scene backward. For bootstrap training at 64-128px with hundreds of splats, choose the low-overhead renderer first; switch to v3 when tile occupancy, wider splats, or 64k-scale gradients make bandwidth/recompute behavior dominate.
- Taichi batch speedup claims are baseline-sensitive. In a fresh `B=4, 128x128, G=128` run, Taichi absolute time was close to the README order of magnitude, but a looped MPS direct-Torch reference was far faster than the README's direct-Torch baseline, shrinking the speedup factor. Compare absolute ms and reference implementation, not just `x` speedups.
- At `1024x1024/G=65536`, Taichi/Metal remains useful as a compatibility path but its backward is not in the same class as pure Metal: measured forward+backward was about `352/1081 ms` for Taichi vs `41/120 ms` for v2 and `20/40 ms` for v3 on sigma `1-5/3-8`.
- The Mac renderer stack now has three distinct proven roles, not one winner: Taichi for compatibility/native batch, fast-mac v2 for tiny bootstrap overhead, fast-mac v3 for large/backward-heavy training. Future renderer selection should be scenario-based, not repo-brand based.
- "Warm" renderer timing means first-use compile/setup effects are excluded, not that rendered splats or outputs are cached. Reusing the same synthetic splats across warmup/timing is fine for kernel timing as long as command sync is correct and no one claims data-randomization coverage.
- Batch support is asymmetric: Taichi has real `[B,G,...]` native batch, while fast-mac v2/v3 are currently single-image APIs looped over frames. This makes Taichi more train-loop convenient than its raw raster speed suggests, and makes a native fast-mac batch API a high-leverage next step.
- The raw MLX/Metal path was valuable as a proof of Metal raster viability, but bridge overhead and dependency split make direct Torch custom ops the better training path. Keep raw MLX in `shader_experiments` for provenance/experiments, not as the main renderer backend.
