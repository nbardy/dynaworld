# Eval SSIM Metric

## Context

The latest run quality looked poor, and the user was worried that the new small token/head init may be too clustered. The trainer already logged `Eval/DSSIM` for the `standard_gs` loss path, but it did not log raw SSIM for the full validation video.

## Change

- Updated `src/train/dynamicTokenGS.py` full-sequence validation logging to compute SSIM once and log:
  - `Eval/SSIM`
  - `Eval/DSSIM`
- These metrics are now logged regardless of the active reconstruction loss type, as long as a full validation video is rendered.

## Validation

- `py_compile` passed for `src/train/dynamicTokenGS.py` and `src/train/losses.py`.
- Synthetic SSIM check produced finite per-image values.
- `git diff --check` passed for touched files.

## Follow-Up Thought

The clustering concern is credible. With `token_init_std=0.01` and `head_output_init_std=0.002`, many decoded attributes start near their biases. We add explicit jitter for position and scale, but RGB starts near `0.5`, opacity starts near the configured bias, and rotations start as identity. If quality stays bad, add init/decode distribution metrics before changing values again.
