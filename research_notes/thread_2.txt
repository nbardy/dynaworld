You are exactly right on the intuition: these models do not need pre-computed 3D splats in the dataset to train. They compute their own splats on the fly during the forward pass, and they are supervised entirely by the 2D video frames.

Because the splatting rasterization process is fully differentiable, feed-forward generators can use a self-supervised photometric rendering loss instead of requiring 3D ground truth.

Here is exactly how models like TokenGS and Lyra 2.0 utilize datasets like DL3DV-10K to learn splat generation:

The Core Training Loop (2D Supervision for 3D Generation)
Instead of comparing output splats to "ground truth" splats, the training loop bridges 2D and 3D using the dataset's calibrated camera poses:

The Forward Pass (Lifting): The model takes one or more sparse image frames as input. The neural network directly regresses the raw parameters for millions of 3D Gaussians (XYZ means, scales, rotations via quaternions, opacities, and Spherical Harmonic coefficients for color).

Novel View Rendering: Using the precise camera poses provided by the dataset (e.g., the COLMAP data in DL3DV), a differentiable rasterizer takes those predicted splats and renders them from a different, held-out viewpoint from that same video sequence.

Loss Calculation: The rendered 2D image is compared against the actual 2D ground-truth frame from the dataset. The model calculates a photometric loss (typically a combination of L1, SSIM, and LPIPS).

Backpropagation: The gradients flow backward from the 2D image loss, through the differentiable rasterizer, and into the network's weights. Over millions of iterations, the model learns exactly how to arrange 3D geometry to satisfy the 2D views.

How TokenGS Approaches This
The recent TokenGS paper specifically identifies a bottleneck in earlier feed-forward 3DGS networks. Standard pixel-aligned architectures usually regress the Gaussian XYZ means as depth values along camera rays. This permanently binds the number of generated primitives to the input image resolution.

TokenGS breaks this constraint. Instead of an encoder-only design that unprojects pixels, it uses an encoder-decoder architecture populated with learnable Gaussian tokens. These tokens cross-attend to compact image features and directly regress the 3D mean coordinates in space. However, the supervisory signal remains exactly the same: it trains completely via that self-supervised 2D rendering loss against the dataset's ground-truth frames, decoupling the splat prediction from the pixel grid while improving robustness to the kind of multi-view inconsistencies you find in raw video datasets.

How Lyra 2.0 Approaches This
Lyra 2.0 uses a two-stage approach aimed specifically at generating massive, explorable 3D worlds from a single image, relying heavily on video diffusion priors:

Video Synthesis with Spatial Memory: It first uses a video diffusion model to generate consistent novel views. To solve the classic "drifting" problem in video generation, it stores per-frame 3D geometry, retrieving relevant past frames to establish dense correspondences and keep the global geometry consistent.

Explicit 3D Reconstruction: It then uses a feed-forward Gaussian reconstructor to "lift" that generated video sequence into an explicit 3D Gaussian representation.

When training systems like Lyra 2.0 on massive static scene datasets, the model is learning two things simultaneously: the generative video prior (how scenes ought to look as a camera moves through them) and the geometric lifting capability (how to accurately cluster Gaussians to represent that sequence), entirely driven by the 2D multi-view data and the camera trajectories.

It is a fascinating idea, and conceptually you are exactly on the right track—but practically, there is a major architectural catch when moving from static to dynamic data.

You cannot simply take a purely static feed-forward model, fine-tune it on a dynamic dataset, and expect it to suddenly understand movement. If you try, you run into severe architectural bottlenecks.

Here is exactly what happens if you try to fine-tune them on dynamic data, and what actually needs to change to make it work.

The "Ghosting" Problem (Naive Fine-Tuning)
If you take a standard architecture designed to output 3D geometry and train it on a dynamic scene (like a person walking across a room), the 2D photometric rendering loss will get completely confused.

The network is only designed to predict static X,Y,Z coordinates. When it tries to minimize the loss against a moving target across different frames, it will attempt to satisfy all frames simultaneously. The result is "ghosting"—the model will either blur the moving person into a semi-transparent smear across their entire path, or spawn hundreds of static, floaty artifacts trying to represent the person at every single frame of the video.

The Fix: The Jump from 3D to 4D
To make these models work for dynamic scenes, you have to modify the architecture to inject a temporal dimension (t). You aren't just predicting 3D Gaussian Splatting anymore; you are predicting 4D Gaussian Splatting.

Instead of just regressing static primitives, the network needs to be restructured to handle motion. There are generally two ways to build this:

Deformation Fields: The model generates a set of "canonical" Gaussians in a neutral pose, and a secondary network (like an MLP) predicts a deformation field. This field applies translations and rotations to the base Gaussians based on the timestamp t.

Flow Matching / Velocity Tracking: The network predicts the initial state of the Gaussians and an accompanying 3D velocity/flow vector for each primitive, allowing them to be forward-warped through time.

How This Affects Specific Architectures
TokenGS-style Models: Because they rely on cross-attending Gaussian tokens to image features, you would need to add temporal attention mechanisms. The tokens would need to track features across a sequence of frames to output dynamic trajectories rather than static points.

Video Diffusion Priors (Lyra-style): Models built on video diffusion backbones actually have a massive head start here. Because video diffusion natively understands spatiotemporal consistency, fine-tuning the generative backbone on dynamic scenes works incredibly well. The challenge just shifts to the lifting stage—the spatial memory module has to track how the generated pixels deform over time to output consistent 4D geometry.

From a systems engineering perspective, if you modify an architecture to output dynamic splats, the real bottleneck becomes the rasterizer. Standard static forward passes won't cut it anymore. You have to update the covariance matrices and positions of millions of Gaussians per frame before rendering, which generally requires writing highly optimized, custom compute kernels just to keep the training loop from grinding to a halt.

Forcing SH0 forces the network to rely entirely on your polynomial geometry movement to satisfy the photometric loss.

def training_step(model, video_dataset, num_samples_per_batch=4):
    \"\"\"
    video_dataset: Contains GT video frames and camera trajectories.
    model: The V-JEPA/Diffusion + TokenGS lifting network.
    \"\"\"
    optimizer.zero_grad()
    loss = 0.0
    
    # Forward pass through backbone to get global splat coefficients
    # This happens ONCE per sequence/video, not per frame.
    splat_coeffs = model.forward_sequence(video_dataset.input_data)
    
    # Sample N random (frame_time, camera_pose, gt_image) tuples
    batch = video_dataset.sample_random_frames(n=num_samples_per_batch)
    
    for t_i, pose_i, gt_img_i in batch:
        # 1. Materialize 3D geometry at specific time t_i
        current_splats = evaluate_4D_splats(splat_coeffs, t=t_i)
        
        # 2. Differentiable Rasterization
        rendered_img = standard_3dgs_rasterizer(
            means3D=current_splats.positions,
            rotations=current_splats.rotations,
            scales=current_splats.scales,
            opacities=current_splats.opacities,
            colors=current_splats.rgb,
            viewpoint_camera=pose_i
        )
        
        # 3. Spatial Photometric Loss
        l1_loss = L1(rendered_img, gt_img_i)
        ssim_loss = SSIM(rendered_img, gt_img_i)
        
        # Accumulate loss
        loss += (0.8 * l1_loss + 0.2 * (1.0 - ssim_loss))
        
    # Optional: Add temporal consistency regularization directly on coefficients
    # e.g., Penalize extreme high-order coefficients to prevent jitter
    reg_loss = compute_coefficient_regularization(splat_coeffs)
    loss += reg_loss
    
    # Backpropagate through the rasterizer, the evaluator, and into the model weights
    loss.backward()
    optimizer.step()
    
    return loss.item()
